# Long-lived PowerShell daemon for GUI-tool PS calls

Every tool in `eos-laptop-agent/tools/` that shells out to PowerShell (clipboard, window, uia, notification, process, partly input) now routes through `lib/ps-daemon.js` - a single long-lived `powershell.exe` child process - instead of paying per-call spawn cost.

## Why

Cold-spawning `powershell.exe` for each tool call has three failure modes that all bite under memory pressure:

1. **Startup cost.** ~500ms per spawn (load profile, .NET assemblies, JIT). For multi-step GUI flows that call PS 10+ times, that's 5s of pure overhead.
2. **System.Web init OOM.** When system memory is high (>85%), `ConvertTo-Json` triggers `System.Web.HttpRuntimeSection` initialization which can throw `"The type initializer for 'System.Web.Configuration.HttpRuntimeSection' threw an exception"` mid-call. Real failure: `win.foreground()` errored exactly this on 2026-05-17 night during dispatch testing.
3. **Set-Clipboard hang.** Under memory pressure, `Set-Clipboard` blocks waiting for the clipboard service to respond. Real failure: `clipboard.write()` returned "clipboard write failed:" (empty stderr) during the same session.

A long-lived daemon eliminates all three because: assemblies load ONCE at daemon start; `Add-Type @"..."@` inline-C# compiles ONCE and is cached forever; no per-call spawn fees; and the daemon's persistent runtime avoids the memory-pressure-during-startup race.

## How (architecture)

- **Daemon process**: `lib/ps-daemon-host.ps1` (auto-generated by `lib/ps-daemon.js`). Reads line-delimited JSON request envelopes from stdin, runs each script in a child runspace with timeout, writes line-delimited JSON response envelopes to stdout.
- **Manager**: `lib/ps-daemon.js`. Spawns the daemon at laptop-agent startup, manages the request queue (one-at-a-time so stdout doesn't interleave), correlates request/response via UUID, auto-respawns on death, heartbeats every 30s.
- **Per-tool helper**: tools call `psd.runOrFallback(script, opts)` which uses the daemon when alive AND falls back to per-call `spawnSync` if the daemon is unavailable. Same return shape both ways.

Request envelope: `{"id":"<uuid>","script":"<ps body>","timeout_ms":5000}`
Response envelope: `{"id":"<uuid>","ok":true,"stdout":"...","stderr":"...","exit_code":0,"elapsed_ms":42}`

## Diagnostic tool surface

- `ps.stats()` - daemon health (alive, pid, requests_served/failed, queue depth, last_error)
- `ps.run({script, timeout_ms})` - raw exec for ad-hoc PS from outside the laptop-agent tool family
- `ps.ensureAlive()` - kick the daemon if dead
- `ps.restart()` - shutdown + respawn

Use sparingly - most callers should route through the per-tool helpers.

## Migration cookbook

Each PS-shelling tool follows the same pattern:

```diff
-const { spawnSync } = require('child_process')
-function runPs(script, timeoutMs) {
-  const r = spawnSync('powershell', ['-NoProfile', '-Command', script], {...})
-  return { exitCode: r.status, stdout: ..., stderr: ... }
-}
+const psd = require('../lib/ps-daemon')
+async function runPs(script, timeoutMs) {
+  const r = await psd.runOrFallback(script, { timeout_ms: timeoutMs || 6000 })
+  return { exitCode: r.ok ? 0 : (r.exit_code || 1), stdout: (r.stdout||'').trim(), stderr: (r.stderr||r.error||'').trim() }
+}
```

Then add `await` at each caller site of `runPs` inside the (already-async) tool functions.

## Don't

- Don't spawn `powershell.exe` directly from a NEW tool. Always go through `psd.runOrFallback`.
- Don't put state in the daemon that persists across requests EXPECTING isolation. Each script runs in its own runspace; variables don't leak (good) but neither do session-level caches you want (so cache at the manager level if needed).
- Don't write tools that read stdin from the daemon directly. The serializing queue in `ps-daemon.js` is the only safe path - bypassing it interleaves stdout and breaks everything.

## Tradeoffs

- Single-request-at-a-time means N concurrent callers serialise. For most GUI flows this is fine (one screenshot at a time is what you want anyway). If we ever need parallelism, the right move is multiple daemon instances, not concurrent stdin writes.
- Fallback to `spawnSync` means we degrade gracefully when the daemon is unavailable, but those calls still hit the original failure modes. So if memory pressure has killed the daemon AND we're using the fallback, we still bleed - just slower than before.
