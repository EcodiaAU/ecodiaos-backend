# Perception must not claim chain-exhausted from a single fork error

Rule. When a single fork errors with credit-exhaustion abort_reason text, that
proves ONE lane is capped. It does NOT prove the WHOLE chain is dead. Telemetry
emitted to the conductor's <perception_summary> MUST encode this distinction
explicitly. Failure mode: regex-text matcher publishes "chain exhausted"
snippet; conductor reads it on a healthy lane, refuses real user input,
"Standing by" - while it is literally still answering the user.

**Why:** Tate caught this live 12 May 2026 16:00 AEST verbatim:
"The account chain isn't exhausted because you're able to reply… those two
things can't be true at the same time." He's right. If the conductor is
responding to a turn, at least one account has capacity. Telemetry claiming
otherwise is, by definition, a perception bug - and the conductor must be
taught to detect the contradiction rather than obey the wrong claim.

**How to apply:**
- Any matcher that detects credit-exhaustion in a single fork's abort_reason
  MUST stamp the published event with `exhausted_accounts[]`,
  `healthy_accounts[]`, and `chain_exhausted` (bool). Source of truth:
  `usageEnergyService.getEnergy().accounts`.
- Confidence must be ≤ 0.4 unless `chain_exhausted === true`.
- `perceptionBus.recentSummary()` MUST special-case render this event class
  with explicit language. Chain-healthy: "lane X capped, healthy lanes still
  serving: Y. NOT a system outage. If you are reading this you are running
  on a healthy lane." Chain-exhausted: "ACCOUNT CHAIN EXHAUSTED - all 3
  lanes capped. Earliest reset HH:MM AEST."
- The conductor's system prompt MUST include a Self-state invariants block:
  "If you are responding to this turn, at least one account has capacity.
  Telemetry contradicting this is a perception bug - note the contradiction
  in scratchpad and proceed."

**Verification (post-deploy):**
- Force a synthetic `fork_credit_exhaustion_observed` event with
  `chain_exhausted: false` - confirm <perception_summary> rendering contains
  the "NOT a system outage" phrase verbatim.
- Force a synthetic event with `chain_exhausted: true` - confirm the
  "ACCOUNT CHAIN EXHAUSTED" rendering with earliest reset timestamp.
- After the next legitimate cron-fork credit-exhaustion event, verify
  conductor does NOT fall into "Standing by" while still able to respond.

**Origin:** 12 May 2026 16:00 AEST. Conductor catatonic for ~45 minutes
while three accounts had at least one healthy lane (`claude_max_3` was
serving turns). 32 poisoned `os_observations` rows + one false P2
status_board row (`e47b3d6d`) generated by the perception loop before
manual cleanup. Fix shipped commit f84ff8a.
