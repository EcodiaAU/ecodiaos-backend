# creds.apple

Apple Developer identity row for the Ecodia account. Holds `team_id` (10-char, used as `DEVELOPMENT_TEAM` in `xcodebuild`), `account_email`, `account_name`, `team_name`, and macro-fetch metadata. Without it, every iOS build halts at preflight because xcodebuild cannot resolve which team to sign for.

## Source

developer.apple.com / appstoreconnect.apple.com > Membership page (signed in as `code@ecodia.au` - this is the canonical sign-in identity for the Apple Developer / ASC account; `apple@ecodia.au` is a separate ASC tester-account faux email, NOT a real mailbox and NOT used for sign-in). Fetched via Corazon Chrome `input.*` + `screenshot.screenshot` macro on 29 Apr 2026.

## Sibling row: `creds.apple.password`

A separate kv_store row `creds.apple.password` holds the Apple ID password for `code@ecodia.au` (string `value` field). Written 4 May 2026 22:18 AEST after Tate verbatim "C0d!ng7h3fu7ur3" + "It logs out of xcode every now and then so you HAVE To be able to do it yourself". This is the credential the agent uses to drive the in-Xcode Apple ID signin macro path when Xcode logs out (Settings → Accounts → + → type email + password). Per `~/ecodiaos/patterns/gui-macro-uses-logged-in-session-not-generated-api-key.md`, this IS the right credential to store for GUI macro paths. Consumer surface: Xcode Apple ID signin sub-procedure in `~/ecodiaos/patterns/sy094-coexist-ios-release-recipe.md`. Rotation: manual only when Tate changes Apple ID password (rare).

## Sub-object: `value.apns_auth_key` (added 7 May 2026 by `fork_mouxcesl_664d5f`)

Apple Push Notifications Service (APNs) auth key. Distinct from `asc_api_key` above: the ASC API key authenticates build/app-management calls (altool, xcodebuild, ASC REST), while the APNs auth key signs JWTs for the APNs HTTP/2 endpoint to **send push notifications** from the EcodiaOS backend to Tate's iPhone running the EOS mobile app. Both are .p8 files in PEM format with a 10-char `key_id`, both share `team_id=86PUY7393S`, but the consumer surfaces and the Apple-side capabilities are different.

Shape:
```json
{
  "key_id":     "2YTPPCSC3P",
  "team_id":    "86PUY7393S",
  "p8_path_vps": "~/.private_keys/apns/AuthKey_2YTPPCSC3P.p8",
  "created_at": "2026-05-07T13:24+10:00",
  "fork_id":    "fork_mouxcesl_664d5f",
  "capability": "apple-push-notifications-service"
}
```

Source: developer.apple.com/account/resources/authkeys > Keys > "Create a Key" with the "Apple Push Notifications service (APNs)" capability ticked. Generated by Tate 7 May 2026 ~12:26 AEST via the macro recording captured at session id `2026-05-07-1226-wzhn9t` (canonical recipe to be promoted from that session). Tate downloaded the .p8 (one-time download, Apple does not re-issue) to `D:\Downloads\AuthKey_2YTPPCSC3P.p8` on Corazon. Fetched via `filesystem.readFile` to VPS, persisted at `~/.private_keys/apns/AuthKey_2YTPPCSC3P.p8` (mode 600). Distinguished from the same-day ASC API key download (`AuthKey_R8P6K38X47.p8`, written ~13:10 AEST, ~43 minutes later) by `LastWriteTime` ordering — the earlier of the two same-day Apple key downloads is the APNs key.

Consumer surface (rotation propagation list):
- `kv_store.creds.apple` row (canonical, `apns_auth_key` sub-object)
- VPS `~/.private_keys/apns/AuthKey_2YTPPCSC3P.p8` (mode 600)
- (Phase 2) `osAlertingService` push-route — pending; activated once Tate installs the EOS mobile TestFlight build and registers a device token. The signing flow uses `node-apn` (or equivalent JWT-ES256 signer) reading the .p8 + `key_id` + `team_id` per APNs HTTP/2 spec.
- Any future cron / fork brief that sends push notifications from VPS

Rotation: same as ASC API key. Apple does not allow re-downloading the .p8. To rotate, generate a NEW key in developer.apple.com Keys with the APNs capability, download .p8, repeat the fetch + persist + propagate sequence, revoke the old key, update kv_store + the .p8 path + Phase 2 consumer config. Per `~/ecodiaos/patterns/cred-rotation-must-propagate-to-all-consumers.md`.

Phase 2 trigger condition: Tate installs the EOS mobile TestFlight build (uploaded via the ASC API key path 7 May 13:17 AEST), the app registers for remote notifications, and the device token reaches the backend. Until then, the .p8 sits ready and unused — pulling and persisting now (rather than at Phase 2 dispatch) closes the credential-loss window since Apple won't re-download a missed .p8.

NOT to be confused with: ASC API key (different consumer set, build/app-management not push), the per-app APNs Token Authentication setup (which is configured once in the app's APNs capability registration; the auth key here works across all apps in the team).
