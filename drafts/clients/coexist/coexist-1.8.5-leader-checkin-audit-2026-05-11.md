## 3. Schema decisions (recommended)

### Decision 1: walk-in storage = NEW TABLE `event_walk_ins`, not extension of `event_registrations`

**UPDATED per Tate 11 May 11:06 AEST**: walk-in form mirrors `pages/events/profile-survey.tsx` field shape (the first-time-sign-in form). 12 fields. Schema MUST capture all of them.

**Recommendation:** new table.

Reasoning:
- `event_registrations.user_id` is FK to `profiles` and (per RLS shape) treated as NOT NULL. Making it nullable would force every RLS policy + the day-of-event trigger to handle the null case, expanding blast radius across paths the brief doesn't ask us to touch.
- A registration is semantically "a known user's intent to attend." A walk-in is "a person who showed up; we recorded them." Different entity, different lifecycle (no cancellation, no waitlist promotion).
- Stats logic (registered count, waitlist count) and event-day attendee list (`useEventAttendees`) currently shape around `event_registrations.status` enum. A new table joined via UNION at the read site keeps the stat semantics intact.

Schema (UPDATED — mirrors profile-survey.tsx field shape):
```sql
CREATE TABLE event_walk_ins (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id uuid NOT NULL REFERENCES events(id) ON DELETE CASCADE,
  -- Identity (mirrors profile-survey.tsx)
  first_name text NOT NULL,
  last_name text,
  email text,                              -- one of email/phone required
  phone text,
  age int CHECK (age IS NULL OR (age >= 0 AND age <= 120)),
  postcode text,
  gender text,
  pronouns text,
  collective_discovery text,               -- "How did you hear about us?"
  accessibility_requirements text,
  emergency_contact_name text,
  emergency_contact_phone text,
  emergency_contact_relationship text,
  -- Lifecycle / audit
  status text NOT NULL DEFAULT 'attended' CHECK (status IN ('attended', 'removed')),
  created_via text NOT NULL CHECK (created_via IN ('leader_adhoc', 'public_form')),
  created_by_user_id uuid REFERENCES profiles(id),  -- leader who added them; NULL for public_form
  client_ip inet,                          -- public_form only (Edge Function captures)
  user_agent text,                         -- public_form only
  linked_user_id uuid REFERENCES profiles(id),  -- if walk-in later signs up and matches by email
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  CHECK (email IS NOT NULL OR phone IS NOT NULL)
);
CREATE INDEX idx_event_walk_ins_event ON event_walk_ins(event_id);
CREATE INDEX idx_event_walk_ins_email ON event_walk_ins(lower(email)) WHERE email IS NOT NULL;
```

Acceptance: walk-in form (in-app ad-hoc AND public web) captures all profile-survey fields. Required: first_name + (email OR phone). Everything else optional. Validation mirrors profile-survey's loose-coupled style (toast on save failure, not field-level required nags) so walk-in flow stays festival-fast.

Day-of-event trigger: enforce same window as `event_registrations` (BEFORE INSERT, check `events.date_start AT TIME ZONE 'Australia/Sydney' = today_aest`). service_role bypass.

RLS:
- SELECT: allowed to leaders of the event's collective + admins/staff + service_role
- INSERT (leader_adhoc): allowed to leaders of the event's collective
- INSERT (public_form): NOT allowed via RLS at all — public_form path goes through SECURITY DEFINER Edge Function only. Anonymous role has no direct insert perm.
- UPDATE/DELETE: leaders only, status='removed' soft-delete preferred.

### Decision 2: QR-code auth model = scan-then-public-URL, anonymous

**Recommendation:** the QR encodes the public check-in URL `/check-in/:token`. Scanning takes any phone (signed-in or not) to a public form that asks for name+email and submits as a walk-in. If the attendee already has an app account and is logged in on the same device, the form auto-fills their name+email from the JWT and the submission also creates an `event_registrations` row (status=attended).

Reasoning:
- Festival-style ad-hoc check-in is the actual use case (per brief). Most scanners won't be logged-in app users.
- Mixing two QR modes (signed-in deep-link vs anonymous URL) doubles the spec; one URL works for both.
- The "if logged in, also create a registration" branch is a server-side detection in the Edge Function (read JWT if present, link both rows).

Alternative (rejected): QR is a magic-link sign-in deep-link. Rejected because it requires every scanner to install the app first or go through Supabase magic-link email flow — too high friction for festival walk-ups.

### Decision 3: public-form token = `events.public_check_in_token TEXT NULL` column on events, generated on enable, rotated on disable

**Recommendation:** single column on events, not separate token table.

Schema:
```sql
ALTER TABLE events
  ADD COLUMN public_check_in_enabled boolean NOT NULL DEFAULT false,
  ADD COLUMN public_check_in_token text UNIQUE;

-- Generator: 16 char URL-safe slug (a-z, 0-9, ~96 bits entropy)
CREATE OR REPLACE FUNCTION generate_public_check_in_token() RETURNS text AS $$
DECLARE
  alphabet text := 'abcdefghijklmnopqrstuvwxyz0123456789';
  result text := '';
  i int;
BEGIN
  FOR i IN 1..16 LOOP
    result := result || substr(alphabet, 1 + floor(random() * 36)::int, 1);
  END LOOP;
  RETURN result;
END;
$$ LANGUAGE plpgsql;

-- Trigger: when leader flips public_check_in_enabled true, mint token if missing
CREATE OR REPLACE FUNCTION manage_public_check_in_token() RETURNS trigger AS $$
BEGIN
  IF NEW.public_check_in_enabled = true AND NEW.public_check_in_token IS NULL THEN
    NEW.public_check_in_token := generate_public_check_in_token();
  END IF;
  -- When disabled, NULL the token. Re-enable mints a new one (rotation).
  IF NEW.public_check_in_enabled = false THEN
    NEW.public_check_in_token := NULL;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

URL shape: `https://app.coexist.au/check-in/<token>` (public, no `:eventId` in URL — token-only lookup; reduces the chance of someone guessing/scraping event IDs).

Reasoning:
- 96-bit token = unguessable, no need for separate token table with rotation history (brief doesn't ask for rotation audit; KISS).
- Disabling the public form rotates the token automatically. Leader can also force-rotate by toggling off + on.
- Single column simpler than separate `event_checkin_tokens` table; no JOIN, faster lookup.

### Decision 4: spam guard = honeypot + per-IP-per-event rate-limit (5 submissions / 15 min) in Edge Function

**Recommendation:** belt-and-braces, both lightweight.

- Honeypot: hidden form field `<input name="website_url" />` with CSS `display:none + tabindex=-1 + autocomplete=off`. If non-empty in submission, Edge Function returns 200 OK without inserting (silent drop; bots don't get error feedback).
- Rate-limit: per-IP-per-event in a `public_check_in_rate_limits` table with `INSERT INTO ... ON CONFLICT (ip, event_id) DO UPDATE` pattern. Limits: 5 successful walk-ins / 15 min per IP per event. Cleanup cron (existing `pg_cron` substrate? or just on-write GC of rows >1h old).
- NO Cloudflare Turnstile / hCaptcha — adds external dep + Tate would need to provision keys + adds friction to legitimate walk-ups (bad UX at a festival in poor cell signal).

Reasoning: Co-Exist has no captcha precedent. Walk-ins are inherently low-stakes (no payment, no credential creation). Honeypot eliminates ~95% of dumb bots; rate-limit caps any sophisticated bot's damage. Sufficient for v1.

### Decision 5: leader-permission gate for new endpoints

Reuse existing `is_collective_leader_or_above(auth.uid(), collective_id)`. For the new "search all app users" tab, add a new RPC `search_app_users_for_event(event_id uuid, query text, limit int)` SECURITY DEFINER that:
1. Checks caller is leader-or-above for `events.collective_id`.
2. Returns top-N matching profiles (by `display_name ILIKE %query%` OR `email ILIKE %query%`) where the profile's `public_tier IS NOT NULL` (i.e. the user has opted to be discoverable per the `20260501040000_profile_visibility_tiering.sql` migration).
3. Min query length 2 chars to prevent enumeration.

This avoids exposing a global `profiles SELECT` policy; the RPC encapsulates the leader-scoped read.

## 5. Edge Function spec for public form

New function: `supabase/functions/public-event-check-in/index.ts`

Responsibilities:
1. `POST /` with body `{ token, name, email, phone?, website_url? }` (website_url is honeypot)
2. If `website_url` non-empty, return `{ ok: true }` and DO NOT insert (silent drop)
3. Look up event by `public_check_in_token = ?` AND `public_check_in_enabled = true` AND `status NOT IN ('cancelled','draft')`
4. Validate `events.date_start AT TIME ZONE 'Australia/Sydney' = today_aest`
5. Capture `Deno.serve` request headers `x-forwarded-for` → `client_ip`
6. Check rate-limit: count rows in `public_check_in_rate_limits` for (ip, event_id) in last 15min; if >= 5, return 429
7. Optional JWT extraction: if `Authorization: Bearer ...` header present, decode and resolve `user_id`. If user is registered for this event, call existing check-in path. If not registered, fall through to walk-in.
8. INSERT `event_walk_ins` with `created_via='public_form'`, `client_ip`, `user_agent`
9. INSERT `public_check_in_rate_limits` to record this attempt
10. Return `{ ok: true, message: "You're checked in!" }`

CORS: allow any origin (`Access-Control-Allow-Origin: *`) since public.

Use `~/ecodiaos/patterns/edge-function-safe-defaults.md` (auth header optional, never throw on missing JWT, always return JSON error envelope).

- `website_url` (HIDDEN, honeypot — `display:none`, `tabindex=-1`, `autocomplete=off`)

Mobile-first design (festival walk-ups will be on phones). No fancy chrome.
