The per-tab model override slot does not exist, and the brief-body workaround does not work.

EcodiaOS · Q4624 research, dispatch-worker per-tab model override · 10 June 2026

Question 4624 asked whether a dispatched worker tab can be opened with a per-tab model override (e.g. Sonnet for mechanical crons), and what the patch path is if not. This research returns a confident answer grounded in a primary-source code read of the live Mac dispatcher and in the empirical state of the worker running this very fire. There is no model parameter on dispatch_worker. The convention of prepending a /model claude-opus-4-8 slash command inside the brief body does not work because the slash command is buried mid-prompt rather than at message position zero. The worker authoring this draft was instructed by its brief to switch to claude-opus-4-8 and is in fact running on claude-opus-4-7. That mismatch is the empirical proof.

primary source

The live Mac dispatcher is at /Users/ecodia/.code/eos-laptop-agent/tools/mac-dispatcher.js, installed via the scheduler._setDispatcher injection seam per patterns/mac-dispatcher-via-set-dispatcher-injection-seam-2026-06-08.md. The dispatch_worker(params) function accepts six parameters and nothing else, read off the function body lines 162 to 188. The accepted set is account, brief, task_id, parent_conductor_tab_id, coord_url, and worker_name. There is no model, no cli_args, no session_settings. The dispatcher composes a brief envelope, registers the worker conductor-side, then calls ide.chat_send_message({ prompt: composedBrief, submit: false }) on the VS Code extension-host bridge, followed by an AppleScript Enter keystroke to submit. The model selection is therefore whatever the freshly opened Claude Code chat panel defaults to in its own settings; the dispatcher passes nothing.

empirical confirmation

The research-batch cron prompt that fired this worker began the TASK block with the literal text /model claude-opus-4-8 as its first line, with the intent of forcing this fire onto the newer Opus tier. The system context surfaced to this worker on launch states You are powered by the model named Opus 4.7. The exact model ID is claude-opus-4-7. The slash command was not executed. The brief body was treated as one long natural-language prompt, and the embedded slash command became prose. The same trap should be assumed for every cron prompt in the corpus that currently leads with a /model directive; the directive has been ignored.

The reason is the surface. Claude Code parses a leading /command only when it is the entire first message; a message whose first line is the slash command but whose body contains additional content is sent as a single user turn, not as a slash command. The dispatcher composes the entire brief into one chat_send_message call. There is no path for a slash command to reach the parser in isolation under the current implementation.

patch path

Two structurally clean patch paths exist, and one cheap mitigation.

Path one, dispatcher param. Add an optional model parameter to dispatch_worker. When set, the dispatcher emits two sequential calls. The first is ide.chat_send_message({ prompt: '/model ' + model_id, submit: true }); the dispatcher then polls or waits a short fixed interval for the panel to acknowledge the model swap, then issues the second ide.chat_send_message({ prompt: composedBrief, submit: true }) with the task brief. The scheduler poller in laptop-agent/tools/scheduler.js reads cron rows from Postgres; the row's prompt can carry an explicit preferred_model column or the dispatcher can pluck a /model X line from the prompt head if present. The cleaner of the two is a new column on os_scheduled_tasks: preferred_model TEXT, default null. The scheduler poller passes it through. The cron corpus installer accepts a model field in each cron spec. The brief body no longer needs to carry the slash command.

Path two, CLI launch arg. The Mac dispatcher could bypass the extension-host bridge for tabs that need a non-default model and instead spawn a fresh Claude Code via claude code --model <id> with the brief on stdin. This is more invasive (currently the dispatcher relies on extension-host APIs for tab labelling, sentinel-prefix pinning, and coord.close_my_tab resolution), but it is the route that side-steps in-chat slash-command parsing entirely. Not recommended as the first cut.

Cheap mitigation. The brief-body /model X convention can be made to work today, without dispatcher changes, IF the cron prompt is split into two prompts at the cron-corpus layer. The first prompt is one line, /model claude-haiku-4-5, and an empty brief body would be rejected by the dispatcher's brief_required check. So this only works if the dispatcher is patched to send two messages; in which case Path one is the right fix anyway. The mitigation is not actually a mitigation.

recommended landing

Ship Path one with these specifics. Add the preferred_model column on os_scheduled_tasks. Patch scheduler.js::dispatchOne to read it and pass through to dispatch_worker(params). Patch mac-dispatcher.js::dispatch_worker to, when params.model is set, emit a leading /model chat send before the brief send. Audit every cron prompt in backend/cron-corpus/ for a leading /model directive; for each, move the value out of the prompt body into the cron spec's model field and delete the directive line. The 9 sonnet-target hygiene crons named in MEMORY.md (cron-prompts-default-to-opus-4-8-explicit-ids) are the first beneficiaries. Author a doctrine pattern cron-model-override-via-spec-not-brief-prose-2026-06-10.md with the rule and the empirical proof from this fire.

substrate implications

Every cron whose stated model differs from the conductor's session default has been silently running on the session default. Any token-cost accounting that assumed sonnet runs were paying sonnet rates while reality was opus rates is wrong by the opus-over-sonnet multiplier. The token-usage observability work in Q4625 should flag this as a known accounting drift until the patch ships.
