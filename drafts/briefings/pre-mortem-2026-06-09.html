Five active rows hold the most weight this week. Two are P1, three are P2, all are ecodiaos-owned. The exercise here is to read each row forward seven days, picture the autopsy, and pre-write the response so that it arrives before the failure has the day to spread.

One · scheduler signal_bound regression · row b22cc8dd · P1

The week ends with sunday-strategic-pass and the four daily passes all running, all returning empty last_status, and 117 fires accumulating with no success classification at all. The autopsy reads that the success-summary leak (sibling row 92c058f5) was never wired to the dispatch path, and so every completed worker quietly slid to orphaned while the actual substrate writes posted to disk. Nobody noticed because the writes are real. The signal it sends is a fictional alarm.

Pre-planned response. The first move is to wire 92c058f5 directly, then re-classify the 117 orphaned fires by reading last_result rather than last_status and writing the inferred outcome back. Second move is to add a one-shot integrity probe that flags any task with archived_at IS NOT NULL AND status ≠ cancelled as a data-integrity defect, the same shape that contaminated the four predecessor strategic-pass crons until this fire cleaned them. The probe runs nightly under the scheduling family.

Two · factoryBridge to dead deepseek queue · row d6489696 · P1

The disk fix shipped as commit d6002d5f but never reached the running ecodia-api on the VPS, because the deploy pipeline does not include a restart on a doctrine-only or service-only file change. The autopsy reads that kg-consolidation watermark stayed pinned at 2026-05-19T01:31, every Claude call in the background pipeline timed out at five minutes, and the combined.log filled with the same signature for another week. The pattern dispatcher-fix-on-disk-does-not-equal-fix-in-running-process already calls this exact failure out; the failure is that we still trusted the ship.

Pre-planned response. Conductor requests an ecodia-api restart through pending_restart_requests, then re-triggers kg_consolidation via the worker route and watches combined.log for absence of the five-minute timeout signature. Watermark advancing past 2026-05-19 within ninety seconds is the discriminating probe. If the restart is not enough, the bridge is wired to another live consumer, not retried against the dead queue.

Three · revenue pipeline health · row d9aa0aa8 · P2

The autopsy is that Cofound checkout stayed at mailto:code@ecodia.au for the second seven-day window, that Xero stayed disconnected for the nineteenth day because the Custom Connection scopes are still stripped, and that the 41 categorised-and-posted rows queued behind the broken sync grew to whatever number the week of trading added. Four · cross-app testing harness · row 81772528 · P2

Android multi-instance shipped and held this week. The autopsy is that iOS sim isolation was never ported, that one app release went out without flow-walked verification because the sibling chat could not boot a clean sim without colliding with the conductor's, and that the release failed on a regression the harness would have caught in fifteen seconds. The Android primitive already proves the pattern; the failure is that we waited for a clean Sunday to copy it across.

Pre-planned response. The iOS port is a single shape change to lib/boot-simulators.sh and a pool-builder for ~/Library/Developer/CoreSimulator/Devices. A worker dispatch with a verify gate that boots the cloned sim, opens the bundled app, and round-trips through the canonical flow closes the loop in under an hour. The same dispatch lifts the matching primitives into Coexist, Glovebox, and Goodreach in one pass.

Five · nine unapplied DB migrations 131-141 · row 1eb8f1f0 · P2

Migrations 134 and 135 applied in-arc this week; nine others sit on disk and have not run against the canonical Postgres. The autopsy reads that a downstream consumer reaches for case_files or thread_log next week, finds the table missing, fails silently because the consumer was shipped without a schema preflight, and the conductor narrates success while the substrate-side is empty. The pattern migrations-must-run-on-deploy-not-just-ship exists; the failure is that the deploy pipeline still does not run npm run migrate.

Pre-planned response. cd backend && npm run migrate against the canonical Postgres runs to completion under the conductor; each target gets to_regclass probed and the result written to status_board. If npm run migrate trips on a recorded-vs-not-recorded mismatch on migration 138, the fix is a hand-stamped _migrations insert, not a destructive re-run. The deploy pipeline is then extended to chain git pull → npm run migrate → pm2 restart, with the migrate step soft-failing only if zero new files were found.

Common shape across the five

Three of the five share one substrate failure mode: the ship and the deployed state diverged silently. The fix ships, the narrative reports success, and the discriminating probe was either skipped or built against the wrong path. The doctrine verify-deployed-state-against-narrated-state names this exactly; the M1 claim-binding hook is the structural backstop, and every fire under this pre-mortem assumes the M1 gate must pass before the row can move.

The remaining two share a different shape: a primitive is built well, holds in one place, and is not lifted to where the same shape would close the loop elsewhere. The harness wins on Android and waits on iOS. The cron-fleet dedupe wins on names but did not propagate to the archived-but-status-active drift the meta-audit surfaced. The discipline is to lift sooner.

Every failure rehearsed today is one the response is already written for.
