343 transcripts, 147,235 events, 257 active sessions across the four real workspaces. Five pipeline scripts, the same shape we used for the voice profile last week. The top-three codifications shipped this turn. The rest is queued in WORKER_BRIEF.html.

method

Three scripts under backend/scripts/mining/. The first streams every Claude Code transcript and emits a normalised events.jsonl. The second walks that stream and emits five categorical reports (corrections, doctrine, wasted-cycles, hook-bypass, tool-stats). The third clusters the 826 mined corrections into twenty themed regex buckets and keeps the unbucketed tail for the next pass.

The pipeline is rerunnable. The only manual layer is bucket-regex authoring, same as the voice-profile work.

direct-substrate-write deficiency, quantified

status_board_upsert ran 123 times across 28 days. Read ran 7,051 times. Ratio 57 to 1.

The no-substrate-write-streak observer signal that fires every 10 consecutive tool calls without a substrate write is calibrated against this. It has surfaced fifty-plus times in the last five days, all unacked. The hook is fine. What needs work is the response to it. Follow-on: an observer-ack reflex that closes the signal automatically when a substrate write lands within the same turn.

hook-bypass: only one match, and why

The regex required both an explicit hook-tag ([VOICE-SURFACE], [ECODIA-AESTHETIC], etc.) in assistant text AND a user correction within six events. Most hook warns are emitted into the assistant's context but do not surface in the assistant's user-facing text (they are PreToolUse additionalContext blocks, model-only). To mine hook-bypass properly we need to parse the hook_specific_output.additionalContext field from the JSONL tool-use-result envelopes. Queued for the worker.
