## Critique #1 — hook:brief-consistency-check: ~8s latency and 45.9% orphaned starts breaks Layer 1 surfacing

**Severity:** 5 (HIGH)
**Failure class:** surfacing_failure
**Neo4j node:** phase-G-audit-2026-05-13/critique-01-brief-consistency-hook-8s-latency-orphaned

**Finding:**
Layer 6 primitive telemetry is now producing data (1,959 rows vs 1 row at the 05-11 audit). Its first actionable signal is alarming: the `hook:brief-consistency-check` hook — the foundation of Layer 1 surfacing — runs at a consistent **~8.1 second average latency** and has a **45.9% orphaned-start rate** (848 started events, 459 ok events, 389 never completed).

The latency is suspiciously consistent across all 3 days with data:
- 2026-05-13: avg=7,952ms (89 ok events)
- 2026-05-12: avg=7,971ms (287 ok events)
- 2026-05-11: avg=8,334ms (80 ok events)

~8 seconds to walk the patterns/ corpus on every dispatch. As the corpus grows past 150+ files, the hook walk time grows linearly. The architecture's hard constraint is that hooks must never add noticeable dispatch latency — 8 seconds violates this in the most visible way possible.

The 45.9% orphaned-start rate (starts with no corresponding ok event) indicates the hook is being killed mid-run on nearly half of all dispatches. When the hook dies, no `[CONTEXT-SURFACE WARN]` is emitted, no `surface_event` is written, and the dispatch proceeds with zero Layer 1 surfacing. The architecture's primary anti-ignorance mechanism is silently absent on ~46% of all dispatches.

The 2026-05-08 baseline shows avg=28,864ms for 3 completed runs — suggesting the problem existed before Layer 6 was activated but was invisible without telemetry.

**Evidence:**
```sql
SELECT primitive_name, status, count(*) as cnt,
  round(avg(duration_ms)) as avg_ms,
  round(percentile_cont(0.95) WITHIN GROUP (ORDER BY duration_ms)) as p95_ms
FROM primitive_perf_event
WHERE primitive_name = 'hook:brief-consistency-check'
GROUP BY primitive_name, status;
-- started: cnt=848, avg_ms=0
-- ok: cnt=459, avg_ms=8168, p95_ms=7998
-- orphaned: 848-459 = 389 (45.9%)
```

**Recommended fix:**
1. Audit `~/ecodiaos/scripts/hooks/brief-consistency-check.sh` for the file-walk logic. Add `--max-depth 1` and file count guard (warn + abort if patterns/ exceeds 200 files).
2. Cache the patterns/ trigger-keyword index in a flat file (regenerated by the nightly index-regen cron, already running at 22:00 AEST). Hook reads the cached flat-file instead of walking every .md on each dispatch.
3. Raise the hook's internal timeout to 15s with an explicit graceful-exit that writes a `TIMEOUT` perf event (distinguishable from crash-exits in the `started` orphan pool).
4. Validate: after fix, avg_ms < 1000ms on new ok rows, orphaned rate < 5%.

**Status board:** P1 (Layer 1 surfacing broken on 45.9% of dispatches), entity_type='infrastructure', tracked under audit batch row.

**Cross-refs:**
- `~/ecodiaos/patterns/context-surfacing-must-be-reliable-and-selective.md` — Layer 1 doctrine; this is the primary mechanism now shown to be unreliable
- `~/ecodiaos/patterns/decision-quality-self-optimization-architecture.md` Layer 6 — the telemetry that surfaces this finding

## Critique #2 — 67.5% of all "unverified" rows are ghost correction-adjacent copies, not genuine dark matter

**Severity:** 5 (HIGH)
**Failure class:** doctrine_failure
**Neo4j node:** phase-G-audit-2026-05-13/critique-02-ghost-unverified-copies-67pct-dark-matter-contaminated

**Finding:**
The `unverified` pool is the architecture's dark matter metric — dispatches with no positive or negative signal, indicating the classification pipeline cannot see what happened. A large unverified pool is expected to shrink as the architecture matures.

The actual picture is worse: **280 out of 415 unverified rows (67.5%) are not genuine dark matter**. They carry the evidence string `correction_text_already_attributed|settled_as_unverified`, meaning:
1. A Tate SMS correction WAS detected within 30 minutes
2. The SMS was attributed to ONE dispatch in the concurrent batch
3. All N-1 other dispatches spawned in the same batch window were stamped `settled_as_unverified` — the correction signal was "used up"

These rows are not dark; they are Schrödinger's outcomes. They co-occurred with a correction event but were not designated its recipient. The batch-spawn pattern (5 forks spawned within seconds of each other) combined with the 30-minute attribution window creates this pattern systematically.

The impact compounds across every downstream metric:
- `verification_rate = (success + correction + failure) / total` = (399+11+78) / 941 = **52%** — this LOOKS healthy but the denominator is inflated by 280 fake unverified rows. True verification rate against genuine dispatches = 488 / (941-280) = **74%** — actually better than reported
- Phase D's `classification_deficit` label on these 280 rows is misleading — they are not classification deficits, they are attribution collateral
- The architecture's "dark matter is shrinking" progress signal is artificially muted

**Evidence:**
```sql
SELECT 
  count(*) FILTER (WHERE evidence LIKE '%correction_text_already_attributed%') as ghost_copies,
  count(*) FILTER (WHERE outcome = 'unverified') as total_unverified
FROM outcome_event WHERE ts > NOW() - INTERVAL '7 days';
-- ghost_copies: 280, total_unverified: 415
-- Ghost fraction: 67.5%
```

**Recommended fix:**
1. In `outcomeInference.js`, when attributing a correction to a dispatch, record the batch context (dispatches within ±60s) and mark co-occurring dispatches with `outcome='correction_adjacent'` and `classification='batch_collateral'` rather than `unverified/classification_deficit`.
2. Exclude `correction_adjacent` rows from the `unverified` pool in all metric computations. They should not count against verification rate.
3. Update the `/api/telemetry/decision-quality` endpoint to surface the ghost_copies count separately so it is immediately visible.

**Status board:** P1, entity_type='infrastructure', tracked under audit batch row.

**Cross-refs:**
- `~/ecodiaos/patterns/outcome-classification-must-distinguish-unverified-from-success.md` — the 4-state model; batch_collateral is a missing 5th state
- `~/ecodiaos/patterns/decision-quality-self-optimization-architecture.md` Layer 4 — the verification_rate metric is currently miscalculated

## Critique #3 — Episode resurface FK null regression: 59.7% still null 2 days after Critique #1 remediation

**Severity:** 4 (MEDIUM)
**Failure class:** doctrine_failure
**Neo4j node:** phase-G-audit-2026-05-13/critique-03-episode-resurface-fk-null-partial-fix

**Finding:**
The 05-11 audit Critique #1 found `dispatch_event_id = NULL` on all 567 `episode_resurface_event` rows, breaking Layer 7's primary KPI. A fix was applied. The result: 787/1,952 rows (40.3%) now have non-null FKs. The fix was partial.

The 1,165 rows still with null FK (59.7%) represent the majority of the table. New rows continue to arrive with null FK — the producer that populates these was fixed in one code path but a second path (accounting for the majority of inserts) was missed.

Since the fix, rows have been arriving at a rate of roughly 400/day (2026-05-11: 457, 2026-05-09: 81, 2026-05-08: 1). Most of those post-fix rows carry null FK, confirming the regression is ongoing, not historical.

The Layer 7 KPI "repeated-failure-after-resurface rate" requires the join chain: `episode_resurface_event.dispatch_event_id → dispatch_event → outcome_event`. This chain is broken for 59.7% of resurface events. Layer 7 observability is effectively still dark for the majority of its data.

**Evidence:**
```sql
SELECT 
  count(*) as total,
  count(*) FILTER (WHERE dispatch_event_id IS NULL) as null_fk,
  count(*) FILTER (WHERE dispatch_event_id IS NOT NULL) as non_null_fk
FROM episode_resurface_event;
-- total=1952, null_fk=1165 (59.7%), non_null_fk=787 (40.3%)
```

**Recommended fix:**
1. Grep all callers of `episodeResurface.recordResurfaces` (or the INSERT equivalent) for whether they pass `dispatch_event_id`. The 05-11 fix caught one caller; find the other(s).
2. Add a NOT NULL constraint or a DB-level warning on `dispatch_event_id` to prevent future regressions — a silent null is the worst kind of FK corruption.
3. For historical null rows, a backfill may be possible if the `ts` field on resurface events overlaps reliably with dispatch_event timestamps (join on `ts` window ±5s as a heuristic).

## Critique #4 — Correction classifier over-applies surfacing_failure: at least 45% of corrections are misclassified

**Severity:** 4 (MEDIUM)
**Failure class:** usage_failure
**Neo4j node:** phase-G-audit-2026-05-13/critique-04-correction-classifier-surfacing-bias

**Finding:**
The 11 correction rows are a positive development. Their classification: 9 `surfacing_failure` (81.8%), 2 `doctrine_failure` (18.2%), 0 `usage_failure`. Manual review of the correction texts against the 4-state model reveals systematic misclassification:

**Correction #1** (correction_text: "fuck me cunt i've told you before. Stop with the 5 forks always rule.")
- Classifier verdict: `surfacing_failure`
- Actual: The `fork-by-default-stay-thin-on-main.md` pattern EXISTS and SURFACES on fork spawns. The conductor applied the pattern incorrectly (slot-fill forks despite explicit doctrine against it). This is **usage_failure**, not surfacing_failure.

**Correction #2** (correction_text: "you didnt send a single message to cowork in that entire time i was gone.")
- Classifier verdict: `surfacing_failure`  
- Actual: `100-percent-autonomy-doctrine-30-apr-2026.md` and the duo-operating-together doctrine both existed and surfaced. The conductor built infrastructure but never exercised the deliverable. **usage_failure**.

**Correction #3** (correction_text: "Okay this is Tate again... using cowork as any sort of a 2nd half of your brain is the wrong move...")
- Classifier verdict: `surfacing_failure`
- Actual: Tate was correcting a fundamental strategic orientation. No pattern existed at the time for "conductor-as-whole-brain vs cowork-as-peer". This is **doctrine_failure** (gap in corpus).

The classifier appears to default to `surfacing_failure` when it cannot find a matching surface_event chain — essentially "if we can't prove it surfaced, assume surfacing failure." This conflates three genuinely different cases: (a) pattern existed and surfaced but was ignored (usage_failure), (b) pattern existed but didn't surface (surfacing_failure), (c) no pattern existed (doctrine_failure).

By blaming surfacing when usage is the real failure, the architecture directs remediation toward Layer 1 trigger-tuning when it should be directing it toward Layer 3 forcing-function escalation. This is the architectural equivalent of treating a compliance failure as an information failure.

**Evidence:**
Manual review of 9 correction rows classified `surfacing_failure`. At minimum 4 of 9 (44.4%) appear to be misclassified based on the correction text content and whether the relevant doctrine existed at dispatch time.

**Recommended fix:**
1. The Phase D classifier must check whether a relevant pattern SURFACED (surface_event join) before defaulting to `surfacing_failure`. If surface_event exists → classify `usage_failure`, not `surfacing_failure`.
2. Add a confidence field to classification. When the classifier cannot find a surface_event chain (making `surfacing_failure` vs `usage_failure` ambiguous), emit `classification='ambiguous_surfacing_or_usage', confidence=0.4` rather than confidently asserting `surfacing_failure`.
3. Introduce human-tagging flow for correction rows: create a status_board row per correction > 7 days old with `classification_tate_tagged IS NULL`, prompting review.

**Status board:** P2, entity_type='infrastructure', tracked under audit batch row.

**Cross-refs:**
- `~/ecodiaos/patterns/decision-quality-self-optimization-architecture.md` Layer 5 — the three failure-mode classes; the classifier is conflating them
- `~/ecodiaos/patterns/outcome-inference-must-seek-evidence-of-failure.md` — the 4-state model doc

## Critique #5 — infrastructure_verified is an undocumented 5th outcome state contaminating all 4-state model metrics

**Severity:** 3 (LOW)
**Failure class:** doctrine_failure
**Neo4j node:** phase-G-audit-2026-05-13/critique-05-infrastructure-verified-undocumented-5th-outcome

**Finding:**
38 rows in the 7-day outcome_event pool carry `outcome='infrastructure_verified'` — a state that does not exist in the canonical 4-state model (`success/correction/failure/unverified`) defined in `~/ecodiaos/patterns/outcome-classification-must-distinguish-unverified-from-success.md` and Layer 4 of the architecture doc.

These rows are classified `verified_clean` by Phase D. Their source is unknown — the architecture doc has no producer definition for this state, no evidence schema, and no metric treatment.

The contamination is currently low (38/941 = 4%) but the absence of doctrine means:
1. Future producers can expand this undocumented state arbitrarily
2. The `success_rate` denominator now includes a 5th state whose semantic relationship to `success` is undefined (is `infrastructure_verified` a subtype of success? A separate class?)
3. Phase D cannot classify them into `usage_failure/surfacing_failure/doctrine_failure` without a definition
4. The architecture's self-consistency guarantee breaks — "the 4-state model" is now false

**Evidence:**
```sql
SELECT outcome, count(*), classification
FROM outcome_event
WHERE ts > NOW() - INTERVAL '7 days'
GROUP BY outcome, classification;
-- infrastructure_verified | 38 | verified_clean
```
No row exists in the architecture doc or pattern corpus that defines `infrastructure_verified`.

**Recommended fix:**
1. Define `infrastructure_verified` in `outcome-classification-must-distinguish-unverified-from-success.md` or explicitly deprecate it and migrate rows to `success` if that is the intent.
2. Audit `outcomeInference.js` to find the code path that produces this state and add inline documentation referencing the canonical model definition.
3. Update the architecture doc's "4-state model" to "N-state model" with an explicit list, or constrain the outcome column via a CHECK constraint to the canonical set.

**Status board:** P3, entity_type='infrastructure', tracked under audit batch row.

**Cross-refs:**
